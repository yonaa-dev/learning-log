# Callback vs Promise

## 핵심 질문
Promise는 왜 등장했는가?

---

## 헷갈렸던 지점
- Promise를 콜백의 문법적 대체라고 생각했다

콜백 방식에서도 비동기 처리는 가능했기 때문에,  
Promise가 단순히 코드를 깔끔하게 만들어주는 문법이라고만 인식하고 있었다.

---

## 구조적 이해
- Promise는 비동기 작업의 **상태(pending / fulfilled / rejected)** 를 명시적으로 표현하는 추상화
- 성공과 실패를 하나의 흐름으로 다루어,  
  에러 처리 방식이 일관되게 유지된다

이로 인해 비동기 로직의 흐름을  
중첩이 아닌 **연결된 상태 변화**로 표현할 수 있다.

---

## 실무 기준
- 비동기 결과를 단순한 값으로 다루지 않는다
- **언제 완료되었는지, 성공인지 실패인지라는 ‘상태’** 를 기준으로 로직을 설계한다
